# Target Yield Confirmation Step

## Problem

Today, after calibration validation, the annual plan is generated by the LLM which silently decides the target yield (`rendement_cible`) in Step 2 of `plan_annuel.prompt.v3.ts`:

- If harvest history ≥ 3 years: `moyenne(3_meilleures_années) × 0.95`
- Otherwise: `potentiel_rendement.central × coefficient_etat_parcelle`

This target drives every NPK dose (`dose = rendement_cible × export_npk[E] + ...`) and the economic estimate, but:

- The farmer never sees the number before the plan is built on it.
- There's no way to override ("je vise 5 t/ha cette année, pas 4.2").
- Overrides (if we had them) aren't validated against what's agronomically possible given the calibration's `yield_potential_min/max`, alternance ON/OFF year, planting system, etc.
- Regenerating the plan would silently change the target — no audit trail.

## Goal

Add an **optional** "Objectif rendement" step between calibration validation and nutrition-option confirmation. Shown automatically only when the calibration is ambiguous (see trigger rules below); otherwise the LLM-computed target stands and the farmer can still open the step manually via an "Ajuster l'objectif" link on the plan page. When shown, displays the suggested target plus the possible envelope from the calibration + varietal referentiel, and lets the farmer either accept or override within a validated band. Persist the confirmed target on the calibration row so the annual plan prompt uses it verbatim instead of recomputing.

## Display Triggers (Optional Step)

The step is **auto-shown** when any of these hold:

- Calibration's `yield_range.wide_range_warning` is true (`yield_potential_max > 2.5 × yield_potential_min`)
- `yield_history.length < 3` AND `phase_age ∈ {juvenile, entree_production}` — young parcels with no history have the widest uncertainty
- Calibration `confidence_level` is `faible` or worse

Otherwise the step is **skipped** — the LLM-computed target remains the system target and a compact "Objectif: X.X t/ha · [Ajuster]" chip appears at the top of the plan page so the farmer can open the step on demand.

## Non-Goals

- No new LLM call — the suggestion is computed deterministically in NestJS using data already in the calibration row (`yield_potential_min/max`, `yield_history` from harvest_records).
- No multi-year targets — one target per calibration/season.
- No per-zone target (whole-parcel only, matches every other AI field).
- No change to yield prediction algorithm inside Step 6 of calibration — we only consume its output.
- No change to the plan regeneration behavior other than reading the confirmed target from persistence.
- No UI for history editing here — harvest records are captured elsewhere.

## Decision Rules

**Suggested target (deterministic, computed before showing the UI)**

| Condition | Formula |
|-----------|---------|
| `harvest_records` with ≥ 3 years of complete data | `moyenne(3_meilleures_années) × 0.95` |
| Otherwise | `yield_potential_central × coefficient_etat_parcelle` where central = (min+max)/2 and coef comes from calibration `health_score` band (0.85 faible → 1.0 bon → 1.05 excellent) |

**Validation envelope (hard — block on submit)**

| Bound | Rule |
|-------|------|
| Lower | `target ≥ yield_potential_min × 0.7` |
| Upper (calibration) | `target ≤ yield_potential_max × 1.1` |
| Upper (varietal × phase) | `target ≤ referentiel.varietes[variety].rendement_kg_arbre[phase_age][max] × planting_density / 1000` |
| Alternance OFF year cap | `target ≤ yield_potential_central × referentiel.ajustement_alternance.OFF` (olivier ≈ 0.6) |

The tightest of the three upper bounds wins. `phase_age` is read from `calibrations.phase_age` (`juvenile | entree_production | pleine_production | maturite_avancee | senescence`). If the variety has `rendement_kg_arbre[phase_age] = null` (senescence/maturite_avancee), fall back to calibration bound only and flag a warning "Parcelle en déclin économique — arrachage recommandé d'après le référentiel".

**Alternance** is read silently from `baseline.alternanceStatus` (calibration output) — no extra UI input. If `indetermine`, skip the OFF-year cap (no assumption).

**Soft warning (yellow banner, not blocking)**

| Condition | Message |
|-----------|---------|
| `target > history_best3 × 1.1` | "Objectif supérieur de X% à votre meilleure moyenne historique — prévision ambitieuse" |
| `target < history_best3 × 0.8` | "Objectif inférieur à votre potentiel historique — êtes-vous sûr?" |
| calibration has `wide_range_warning` (max > 2.5 × min) | "Calibrage à large fourchette — objectif à affiner après une saison" |

**If user submits outside hard band** → reject with the binding constraint named ("Objectif 8 t/ha dépasse le potentiel max × 1.1 = 6.6 t/ha"). No silent clipping.

## Impact

- Farmer sees and owns the number that drives all downstream doses and costs.
- Plan regeneration is idempotent on the confirmed target — no drift between generations.
- NPK dose calc trusts a stored input instead of a black-box LLM derivation.
- Audit trail: `confirmed_by`, `confirmed_at`, `source: suggested|override`.
- Aligns exactly with the existing `ai_nutrition_option` pattern — minimal cognitive load to add and maintain.

## Persistence Choice

Store on the **calibrations** row (not parcels):

```
ALTER TABLE calibrations ADD COLUMN target_yield_t_ha NUMERIC(6,2);
ALTER TABLE calibrations ADD COLUMN target_yield_source TEXT
  CHECK (target_yield_source IN ('suggested', 'user_override'));
ALTER TABLE calibrations ADD COLUMN target_yield_confirmed_at TIMESTAMPTZ;
ALTER TABLE calibrations ADD COLUMN target_yield_confirmed_by UUID REFERENCES auth.users(id);
```

Rationale: the target is calibration-version-bound. Recalibration resets it (as it should — new potential, new target). Parallels `ai_nutrition_option` but scoped correctly: nutrition option lives on parcel because it's also consumed outside calibration context; target yield is purely a plan-generation input tied to the current calibration snapshot.

## Flow Change

```
Before:
  calibration validated → nutrition-option confirm → plan generate (LLM picks target)

After:
  calibration validated → target-yield confirm → nutrition-option confirm → plan generate (target injected)
```

Annual plan prompt change: when `parameters.targetYieldTHa` is provided in the input, prepend "Rendement cible IMPOSÉ par l'utilisateur — ne pas recalculer à l'étape 2. Passer directement à l'étape 3."

## Re-confirmation After Plan Is Validated/Active

When the farmer changes `target_yield_t_ha` on a calibration whose latest annual plan is already `validated` or `active`:

- The plan is **not** auto-regenerated.
- Plan page shows a persistent yellow banner: "Objectif rendement modifié (X.X → Y.Y t/ha). Le plan actuel reflète encore l'ancien objectif. [Régénérer le plan]".
- The plan row gets a `plan_data.target_yield_drift = { previous, current, changed_at }` marker so the banner is idempotent across reloads.
- Regenerate is the existing `regeneratePlan` flow, which will now consume the new persisted target via prompt injection.

## Decisions

| # | Question | Decision |
|---|----------|----------|
| 1 | Mandatory vs optional | **Optional** — auto-shown only on ambiguous calibrations (wide range, no history + young, or low confidence); always accessible via "Ajuster l'objectif" chip on plan page |
| 2 | Alternance input | **Trust `baseline.alternanceStatus` silently** — no extra UI, skip OFF-cap if `indetermine` |
| 3 | Re-confirm after plan active | **Manual regenerate with yellow banner** — no auto-regen, drift marker on `plan_data` for idempotence |
